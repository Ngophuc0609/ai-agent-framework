# Source Code Handover Workflow

## Language Policy

### AI Execution Language

All workflow instructions, agent prompts, rules, schemas, artifact contracts, validation reports, script messages, status labels, review checklists, coverage reports, provenance scan reports, and negative-evidence reports MUST be written in English.

Agents 1-8 produce English canonical findings, inventory reconciliation, verification, coverage reports, conflict reports, readiness decisions, and `STATUS.md` updates.

### Final Documentation Language

Agent 9 produces Vietnamese final documentation in `.ai/runs/source-code-handover/<run_id>/final/`. Published `docs/` output generated by this workflow MUST also be Vietnamese.

Agent 10 validates:
- Final docs are Vietnamese.
- Non-code prose in final docs is not primarily English.
- Technical identifiers, paths, commands, Evidence IDs, API routes, config keys, JSON keys, database names, and code blocks are preserved.
- Intermediate artifacts remain English.
- No mixed-language instruction policy overrides this workflow.

English intermediate artifacts are not deliverables for Vietnamese developers and must not replace Agent 9 final docs.

## Execution Isolation Policy
Isolation mode hierarchy:
1. `subagent-isolated-worktrees`
2. `isolated-sequential-sessions`
3. `blocked-no-isolation-capability`

**Forbidden Legacy Modes**:
- `single-runtime-sequential-fallback`
- `single-session-multi-role-execution`
- `memory-only-agent-handoff`
- `implicit-agent-output`
- `direct-final-handbook-without-artifacts`

**No Isolation Capability Fallback**:
If the runtime cannot spawn subagents and cannot create isolated sessions:
- MUST STOP before Agent 1.
- Log `blocked-no-isolation-capability` in STATUS.md.
- MUST NOT run Agent 9.
- MUST NOT publish to `docs/`.
- MUST NOT mark `Partial`.
- Generate block report stating exact reason.

## STATUS.md Contract
Must create `.ai/runs/source-code-handover/<run_id>/STATUS.md` with:
- Run ID
- Source Commit
- Git Remote
- Branch
- Execution Mode
- Started At
- Current Phase
- Coordinator Session ID
- Table mapping: Agent | Session ID/Subagent ID | Worktree | Canonical Artifact | Status | Gate
If missing `session_id`, `subagent_id`, or `worktree`, Isolation Verified CANNOT be `yes`.

## Run Initialization

Preferred initialization command:

```bash
RUN_ID="run-$(date +%Y%m%d-%H%M%S)"; ./.ai/scripts/init-source-code-handover-run.sh "$RUN_ID"
```

This creates the canonical run directory, `STATUS.md`, metadata, inventory, evidence, findings, verification, drafting, final, validation, and publish directories.

For Cline, use the command above as a single `execute_command` call with `requires_approval=false`. Do not manually create the run tree through repeated ad hoc `mkdir` commands. If Cline reports a malformed `execute_command` call, retry once with both required fields (`command` and `requires_approval`); after that, stop and report the blocked tool-call limitation instead of looping.

## Phase 0: Preflight + Deterministic Discovery
Coordinator MUST create `.ai/runs/source-code-handover/<run_id>/inventory/` before Agent 1 runs.
Mandatory JSON files:
- projects.json, entry-points.json, configuration-files.json, dependencies.json, dbcontexts.json, dbsets.json, entities.json, entity-configurations.json, migrations.json, seed-data.json, raw-sql.json, sql-metadata.json, controllers.json, routes.json, api-contract-sources.json, hubs.json, realtime-events.json, background-jobs.json, redis-cache.json, queues.json, integrations.json, docker-services.json, ci-cd-files.json, tests.json, runtime-artifacts.json.

Inventory Metadata Required:
`run_id`, `source_commit`, `generated_at`, `source_roots`, `discovery_method`, `status` (complete | partial | scan_failed), `items`, `limitations`.
Count 0 is ONLY allowed if scan succeeds but finds nothing. If scan fails, mark `scan_failed`.

Deep Inventory Requirements:
- `entities.json` MUST include entity class, source path, mapped table/schema when known, key fields, status/type/state fields, and mapping source.
- `sql-metadata.json` MUST include table, column, PK/FK/index/constraint metadata when available; if live DB metadata is unavailable, it MUST record migration/source-derived coverage plus a limitation.
- `routes.json` MUST include controller/action or endpoint handler, route prefix, HTTP method, auth markers, request DTO/model, response DTO/model when source-visible, and source path.
- `api-contract-sources.json` MUST include request/response field inventories from Swagger/OpenAPI/Postman/tests/source DTOs when available.
- `background-jobs.json` MUST include registration source, trigger, schedule, handler, producer/consumer, queue/storage, retry/failure hints, and source path.
- `realtime-events.json` MUST include hub/socket class, mapped route, producer, event name, payload model/fields, group/user mapping, client handler, and source path when available.
- Missing deep inventory fields MUST be explicit `unknown` or `not_found_after_scan` with search patterns/commands, not silently omitted.

## Discovery And Evidence Responsibility Split

Agents 1-5 are broad physical discovery agents. They MUST run in separate isolated sessions/worktrees and scan the current project files to create structured inventories for their assigned domains. Their priority is breadth and completeness: list every discovered component, path, route, table, key, job, config, integration, client, risk, and gap with enough source location detail for later verification.

Agents 1-5 MUST NOT be treated as final evidence authorities. Their findings are discovery candidates. They MAY use `DISC-*` discovery IDs and `[DISCOVERED]`, `[INFERRED]`, `[UNVERIFIED]`, `[CONFLICT]`, `[NOT_APPLICABLE]`, or `[BLOCKED]` statuses. They MUST NOT promote shallow scan results to final `[CONFIRMED]` behavior claims.

Agent 6 is responsible for source/symbol claim verification. Agent 6 MUST re-run targeted tools against the current physical repository, prove or reject Agent 1-5 discovery candidates at file/range/symbol level, and populate source/symbol evidence.

Agent 7 is responsible for cross-layer flow and conflict verification. Agent 7 MUST trace flows across API/auth/service/repository/SQL/Redis/job/external/frontend/config domains and find contradictions across domain findings.

Agent 8 is responsible for safety/build/test/runtime/ops evidence. Agent 8 MUST run safe checks, collect runtime/ops proof when available, perform secret leakage review, and record limitations.

Agent 9 formats final Vietnamese developer documentation from triangulated Agent 6-8 evidence only. Agent 9 MUST NOT add new technical claims and MUST NOT promote Agent 1-5 discovery candidates without Agent 6-8 verification.

Agent 10 independently validates final docs, evidence IDs, physical provenance, language, coverage, safety, source freshness, and publish readiness.

## Evidence Store Contract
Coordinator MUST create `.ai/runs/source-code-handover/<run_id>/evidence/` before Agent 1 runs.
Mandatory evidence files:
- `tool-runs.jsonl`
- `evidence-manifest.json`
- `focused-slices.json`
- `symbol-reference-map.json`
- `data-flow-map.json`
- `sql-metadata.json`
- `api-contract-sources.json`
- `runtime-artifacts.json`
- `tool-limitations.json`

Agents 1-5 SHOULD append tool attempts when the runner provides shared evidence-store access, but their main contract is complete physical discovery in their own findings files. Agents 6-8 MUST complete and normalize the evidence store before Agent 9 writes final docs. Agent 9 MUST NOT write final docs until Agents 6-8 confirm the evidence store is complete enough for the target readiness level.

## Final Documentation Quality Checklist
Agent 9 and Agent 10 MUST apply `.ai/rules/08-source-code-handover-quality-checklist.md`.
The checklist is part of the publish gate, not advisory guidance.
Agent 1-5 MUST write findings with `.ai/templates/source-code-handover/agent-findings-template.md`.
Agent 9 MUST write final documents with `.ai/templates/source-code-handover/final-document-template.md`.
Evidence store files SHOULD follow `.ai/templates/source-code-handover/evidence-store-template.md`.
These templates are mandatory so every project gets the same document structure and review surface.

## Pipeline Phases
Phase 1: Agent 1-5 Broad Physical Discovery (English inventory findings from isolated sessions).
Phase 2: Agent 6 Source/Symbol Claim Verification (English source/symbol evidence).
Phase 3: Agent 7 Cross-Layer Flow And Conflict Verification (English flow/conflict evidence).
Phase 4: Agent 8 Safety/Build/Test/Runtime/Ops Evidence Verification (English safety/runtime evidence).
Phase 5: Agent 9 Final Documentation Assembly (Vietnamese final docs in `final/`).
Phase 6: Agent 10 Independent Publish Validation (English validation reports in `validation/`, including language validation).
Phase 7: Revision if Agent 10 REJECTS.
Phase 8: Final publish (copies `final/` to `docs/`).

## Publish Gate Rule
Docs from `.ai/runs/.../final/` MUST NOT be copied/published to `docs/` unless Agent 10 outputs `PASS` in `final-verdict.md`.
If `REJECT_REQUIRES_REVISION`, `docs/` must not be overwritten.
